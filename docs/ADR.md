# Architecture Decision Records

## 철학
MVP 속도와 정서적 완성도를 동시에 노린다. 외부 의존성은 최소화하되, "도트 방"이라는 비주얼 정체성만큼은 타협하지 않는다. 작동하는 최소 구현을 선택하고, 범용 라이브러리(달력/상태관리 등)는 필요해지는 순간 도입한다.

---

### ADR-001: Next.js 15 App Router
**결정**: Next.js 15 App Router + Server Components 기본, Client Component는 인터랙션 지점에만.
**이유**: 라우트 핸들러로 알라딘 프록시를 같은 레포에서 처리할 수 있고, 서버 컴포넌트로 초기 로드 비용을 줄일 수 있다. Bun과 Next.js 15 조합은 개발·빌드가 빠르다.
**트레이드오프**: Pages Router만 아는 협업자에겐 학습 비용. Server/Client 경계 설계를 잘못하면 "use client"가 트리 전체에 전염될 수 있음.

### ADR-002: Supabase 단일 백엔드 (Auth + Postgres + Storage)
**결정**: 인증·DB·파일을 모두 Supabase로 통일. `@supabase/ssr`로 쿠키 기반 세션.
**이유**: 백엔드 1인분으로는 Firebase/직접서버 대비 SQL 데이터 모델이 유리하고, RLS로 클라이언트 노출 키 상황에서도 멀티테넌시 보안을 단순화할 수 있다.
**트레이드오프**: 벤더 락인. 오프라인 동기화는 v1.1로 미룸. 서버리스 쿼리 지연이 간혹 눈에 띌 수 있음.

### ADR-003: 비회원은 IndexedDB 로컬 전용 (v1.1까지 서버 업로드 없음)
**결정**: 비회원 데이터는 `idb-keyval`로 IndexedDB에만 저장. 서버로 전송하지 않는다. "로컬→서버 동기화"는 v1.1에서 명시 동의 후 추가.
**이유**: 가입 장벽 없이 체험하게 하고 싶지만, 식별되지 않은 사용자의 독후감을 서버에 쌓으면 개인정보/이관/삭제 책임이 커진다. 스코프 최소화를 우선.
**트레이드오프**: 기기 변경 시 데이터 유실. 사용자에게 "로그인하면 안전" 배너로 고지.

### ADR-004: 책 메타데이터는 알라딘 Open API (서버 프록시 경유)
**결정**: `lib/aladin.ts`를 통해 `app/api/books/{search,isbn}/route.ts`에서만 호출. TTB 키는 `ALADIN_TTB_KEY` 환경변수.
**이유**: 국내 도서 커버리지·표지 이미지·출판사 데이터 품질이 가장 좋다. 구글 북스는 국내서 메타가 약하고, 카카오/네이버 책은 각각 승인·센터키가 필요해 초기 진입이 느리다.
**트레이드오프**: TTB 키는 개인 개발자 약관이 있음. 일일 호출 제한에 대한 대비가 필요(MVP는 무시).

### ADR-005: 도트 아트는 정적 PNG 스프라이트 레이어 합성
**결정**: 방 배경 + 곰 + 다이어리 + 책더미 + 창문 + 램프를 각각 PNG로 제작해 절대좌표로 겹친다. `image-rendering: pixelated`.
**이유**: SVG 픽셀 아트는 코드로 생성하면 공수 과다, 런타임 캔버스는 초기 구현 비용과 접근성 비용이 크다. 정적 PNG는 디자이너가 교체하기 가장 쉽고, hitbox는 투명 `<button>` 오버레이로 충분하다.
**트레이드오프**: 곰 idle 애니메이션을 넣으려면 스프라이트 시트가 추가로 필요. 크기 변경 시 에셋을 재생산해야 한다.

### ADR-006: 픽셀 폰트 Galmuri11 셀프호스팅
**결정**: Galmuri11(OFL) woff2를 `public/fonts/galmuri/`에 두고 `@font-face`로 로딩. 영문/숫자는 폰트 내장분 사용.
**이유**: 웹폰트 서비스(구글 폰트 등)에 한글 픽셀 폰트가 드물고, 셀프호스팅이 캐시·프라이버시·CLS 모두 유리하다.
**트레이드오프**: 폰트 업데이트를 수동으로 해야 함. 라이선스(OFL) 고지 의무.
